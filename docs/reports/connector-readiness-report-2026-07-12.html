How support is inferred from code

The documentation should use the catalog as the source for connector identity, config field spelling, source modes, and target declaration. The flow below is intentionally narrower than “this connector works in production.”

01 Connector spec Upstream connector spec.json, fields, and metadata.
02 Catalog assembly Normalizes fields and merges declared/derived capability signals.
03 Bundled JSON index.json plus one <id>.json per entry.
04 Offline validate Checks known mode, config type, and enum values where the catalog is authoritative.
05 Runtime connection Not implemented in this build; cannot establish live support.

Primary code locations

core/core-catalog/src/main/resources/catalog/index.json lists the 77 catalog IDs.

core/core-catalog/.../catalog/<id>.json supplies modes, sink.capable, config, and provenance.

CyntexCatalog.java loads those immutable resources for the wizard, validate, explain, and completion paths.

How the catalog is produced

tools/catalog-assembler walks a pinned connector checkout and normalizes its specs.

SpecNormalizer.java flattens Formily connection fields; ModeResolver.java resolves declared or derived modes.

Each entry carries provenance such as connector repository SHA and spec path. It is a build-time projection, not a runtime certification record.

What the latest code proves—and does not prove

Code-backed today

Catalog Known IDs, normalized field names/types/defaults/options, declared or derived source modes, and the target capability flag.

Offline validation CapabilityRules.java rejects a known, invalid trusted mode and invalid typed/enum configuration values.

Adapter seam The PDK adapter can load an isolated connector class and read registered capabilities without opening a database connection.

Not established by this code baseline

Connection test UnavailableConnectionProbe.java explicitly returns NOT_IMPLEMENTED.

Runtime success No connector in this report has been proven against live credentials, network routes, permissions, read/write behavior, TLS, or performance.

Offline limits Unknown connector IDs and config keys are deferred; required-field presence is also server-side. A successful local check is not a valid connection.

Implementation naming is still behind the planned product rename.

Latest product code still accepts cyntex/v1, *.cyn.yml, and the cyntex CLI. Reader documentation intentionally uses the planned TapState contract. Before customers execute the docs against this codebase, the product rename must land or the docs must carry an explicit compatibility transition.

Recommended next work

Sequence the work by truthfulness risk first, then broaden coverage and maturity only with executable evidence.

P0 · Resolve the four documentation/catalog deltas

Reconcile PostgreSQL’s blank source/target/mode contract; establish catalog or server evidence for Oracle and SQL Server; choose the canonical GitHub ID case. Do not automatically downgrade a guide merely because the offline catalog is incomplete.

P0 · Decide the executable naming contract

Land the TapState CLI, file extension, schema version, and workspace-variable changes in product code—or document a temporary compatibility contract. The current executable code and reader examples differ.

P1 · Add a real connector probe and a small verification matrix

Replace the unavailable probe with engine-backed connection testing. Start with GA connectors: MySQL, PostgreSQL, Oracle, SQL Server, MongoDB, MongoDB Atlas, TiDB, ClickHouse, Doris, and Kafka. Record reproducible environment, server version, command, and outcome.

P1 · Make catalog-to-doc drift a CI failure

Generate a machine-readable comparison of published IDs, modes, roles, fields, and defaults against the bundled catalog. Allow an explicit exception list only where a server-side contract is intentionally authoritative.

P2 · Triage the 33 deferred upstream pages

Use the existing coverage record to choose candidates by customer value, global applicability, and a usable current contract. Treat cloud variants as base-guide coverage unless their authentication, networking, CDC, or compatibility path truly differs.
