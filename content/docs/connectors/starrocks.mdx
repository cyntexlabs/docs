## Before you begin

Prepare a user with read or write access to the selected database and verify the Frontend SQL and HTTP endpoints. For targets, decide key model, table type, distribution, and partitioning before production writes.

## Create a connection

```yaml title="tapstate-work/source/analytics_starrocks.tapstate.yml"
version: tapstate/v1
kind: source
id: analytics_starrocks
connector: starrocks
config:
  host: fe.starrocks.internal
  port: "9030"
  starrocksHttp: http://fe.starrocks.internal:8030
  database: analytics
  user: ${STARROCKS_USER}
  password: ${STARROCKS_PASSWORD}
  useHTTPS: false
mode: snapshot
tables:
  - events
```

For a target connection, omit `mode` and `tables`.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

## Limitations

- The catalog declares Snapshot only; it does not declare CDC or DDL behavior.
- The upstream target baseline references Stream Load, but the current catalog does not publish its tuning or delivery semantics.
- Confirm HTTPS behavior, certificate handling, target table design, and type mapping in staging.

## Reference

| `useHTTPS` | No | `false` | Catalog HTTPS boolean. |
