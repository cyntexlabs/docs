## Before you begin

### Source

1. Create or select a Lark/Feishu self-built application and copy its App ID and App Secret.
2. Set the application's visibility scope so intended content owners are included.
3. Grant the least document, drive, or wiki permissions that the intended read requires.
4. Review the platform's API frequency limits before scheduling downstream reads.

![Locate application credentials in Lark or Feishu Open Platform](/images/connectors/lark-app-credentials.png)

The connector also recommends publishing the app and assigning any required document or wiki administrator role. Confirm the exact permission model with the tenant administrator before a production run.

## Create a connection

```yaml title="tapstate-work/source/knowledge_lark.tapstate.yml"
version: tapstate/v1
kind: source
id: knowledge_lark
connector: lark-doc
config:
  app_id: ${LARK_APP_ID}
  app_secret: ${LARK_APP_SECRET}
mode: api
```

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

Local validation checks the resource envelope, API mode, and recognized field names. It does not authenticate to Lark/Feishu, test app publication or permissions, enumerate documents, or retrieve content.

## Limitations

- TapState supports an API source only. It does not declare Snapshot, CDC, target writes, or DDL behavior.
- Document visibility, tenant policy, rate limits, and API responses are enforced by Lark/Feishu at runtime.
- Do not commit App Secret values in a `.tapstate.yml` file.

## Reference

|---|---|---|
| `app_id` | Yes | Self-built application App ID. |
| `app_secret` | Yes | Application secret; inject through an environment variable. |
