## Before you begin

### Source

<SourceModeTabs label="Choose the PolarDB for PostgreSQL source read mode">
<SourceModeTab value="snapshot">

Use **Snapshot** to read the selected schema and tables once:

```sql
GRANT CONNECT ON DATABASE database_name TO tapstate_reader;
GRANT USAGE ON SCHEMA schema_name TO tapstate_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA schema_name TO tapstate_reader;
```

</SourceModeTab>
<SourceModeTab value="snapshot-cdc">

Use **Snapshot + CDC** to follow ongoing changes after the initial copy. Start with the Snapshot grants, then grant replication access and prepare logical decoding:

```sql
ALTER USER tapstate_reader REPLICATION;
```

Configure logical WAL, enough replication slots and senders for concurrent pipelines, and a `pg_hba.conf` rule that allows the runtime network. For `pgoutput`, make sure the required publication exists and selected tables have a suitable replica identity.

</SourceModeTab>
</SourceModeTabs>

### Target

Grant a dedicated writer access to the intended destination schema:

```sql
GRANT CONNECT ON DATABASE database_name TO tapstate_writer;
GRANT USAGE, CREATE ON SCHEMA schema_name TO tapstate_writer;
GRANT SELECT, INSERT, UPDATE, DELETE, TRUNCATE
  ON ALL TABLES IN SCHEMA schema_name TO tapstate_writer;
GRANT USAGE, SELECT, UPDATE
  ON ALL SEQUENCES IN SCHEMA schema_name TO tapstate_writer;
```

## Create a connection

```yaml title="tapstate-work/source/orders_polardb_pg.tapstate.yml"
version: tapstate/v1
kind: source
id: orders_polardb_pg
connector: polar-db-postgres
config:
  host: pc-abc.pg.polardb.rds.aliyuncs.com
  port: "5432"
  database: orders
  schema: public
  user: ${POLARDB_PG_READER_USER}
  password: ${POLARDB_PG_READER_PASSWORD}
  logPluginName: pgoutput
  timezone: "+00:00"
mode: cdc
tables:
  - orders
```

Use `snapshot` for a one-time copy. For a target, omit `mode` and `tables` and use writer credentials.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

Run a staging snapshot and verify an insert, update, and delete through the selected decoding path before production.

## Limitations

- CDC depends on retained WAL and an available replication slot. If the required position is no longer available, take a new snapshot.
- Tables without a usable replica identity can produce incomplete keys for updates and deletes.
- The documented connector options do not establish a TLS, connection-test, or DDL-control workflow; configure those only through the deployment capabilities provided to you.

## Reference

| Field | Required | Default | Use |
|---|---:|---|---|
| `host`, `port` | Yes | — | PolarDB PostgreSQL endpoint and listener port. |
| `database`, `schema` | Yes | — | Database and schema to discover. |
| `user`, `password` | User no | — | Database credentials. |
| `extParams` | No | — | Additional connection parameters. |
| `logPluginName` | Yes | `pgoutput` | Decoding method: `pgoutput`, `decoderbufs`, or a supported `wal2json` variant. |
| `timezone` | No | Empty | UTC offset for temporal conversion. |
