  The connector ID is case-sensitive: `GitHub`. The resource requires `code`, but local validation does not exchange or verify that authorization value.

## Before you begin

### Source

Decide which organization and repositories the integration may read, then arrange the required GitHub authorization with an administrator. Keep the resulting credential out of source control and grant only the scopes needed for the objects you intend to ingest. For the browser authorization sequence, see GitHub's [OAuth app authorization documentation](https://docs.github.com/en/apps/oauth-apps/building-oauth-apps/authorizing-oauth-apps).

## Create a connection

```yaml title="tapstate-work/source/engineering_github.tapstate.yml"
version: tapstate/v1
kind: source
id: engineering_github
connector: GitHub
config:
  code: ${GITHUB_AUTHORIZATION_CODE}
mode: api
```

`code` is the complete connection contract. Its exchange, refresh, and scope behavior must be confirmed with the deployed runtime before production use.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

The local check validates resource shape and API mode. It does not open a browser, authenticate to GitHub, check scopes, discover repositories, or read issues and pull requests.

## Limitations

- TapState supports an API source only; target writes, Snapshot, CDC, DDL, and rate-limit behavior are not declared.
- The connector describes an interactive authorization flow, which is not presented here as a TapState UI or CLI capability.
