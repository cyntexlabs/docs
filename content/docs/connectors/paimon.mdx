## Before you begin

### Source

The TapState catalog declares Snapshot and CDC reads, but the upstream connector guide documents Paimon as a target. Before using it as a source, verify table discovery, the initial position, checkpoint recovery, and schema handling with the exact connector package supplied by your deployment.

### Target

Choose the warehouse location and grant the runtime read/write access only to that path:

- S3 or compatible storage: use `s3://bucket/path` and an endpoint, access key, secret key, and region appropriate for that service.
- HDFS: use `hdfs://namenode:port/path` and a user that can create and modify files below the warehouse path.
- OSS: use `oss://bucket/path`; the endpoint should not include the bucket name.
- Local or NFS: use a path visible from every runtime instance that may execute the pipeline, with enough space for data and compaction work files.

Define a primary key for upserts and deletes. Use partitions for large tables when they match query and retention patterns, and agree on compaction ownership before the first write.

## Create a connection

```yaml title="tapstate-work/source/lake_paimon.tapstate.yml"
version: tapstate/v1
kind: source
id: lake_paimon
connector: paimon
config:
  warehouse: s3://company-lake/paimon
  storageType: s3
  s3Endpoint: https://s3.amazonaws.com
  s3AccessKey: ${S3_ACCESS_KEY}
  s3SecretKey: ${S3_SECRET_KEY}
  s3Region: us-east-1
  database: default
mode: snapshot
tables:
  - orders
```

Use `mode: cdc` only after staging it. For target-only use, omit `mode` and `tables`.

## Validate the configuration

## Limitations

- The catalog does not define catalog/metastore fields, CDC checkpoints, schema evolution, compaction, or write conflict semantics.
- `storageType` selects conditional fields; provide only the matching S3, HDFS, or OSS inputs.
- The upstream baseline supports DML but not runtime schema evolution. Create or evolve table schemas through an approved external process unless the deployed connector contract states otherwise.
- Paimon has primary keys but no secondary indexes. Wide or frequently changing keys affect write layout and compaction.
- A delete needs a complete before image when soft-delete handling converts it into an update marker. Without it, non-key columns can become `null`.
- Upstream tuning values for batch size, write threads, and compaction were not measured in TapState and are intentionally not copied as defaults.
- Keep storage secrets outside committed resource files.

## Reference

| Storage | Fields |
|---|---|
| Common | `warehouse`, `storageType`, `database` (default `default`) |
| S3 | `s3Endpoint`, `s3AccessKey`, `s3SecretKey`, `s3Region`, `s3Properties` |
| HDFS | `hdfsHost`, `hdfsPort` (default `9000`), `hdfsUser` (default `hadoop`) |
| OSS | `ossEndpoint`, `ossAccessKey`, `ossSecretKey` |

`storageType` accepts `s3`, `hdfs`, `oss`, and `local`.

The reused connector baseline supports Paimon 0.6+ data types. Verify precision-sensitive mappings against the table's file format—for example, Parquet logical date representation—before writing production data.
