## Before you begin

Choose the warehouse location and grant the runtime least-privilege storage access. Confirm the Paimon catalog/metastore arrangement, table primary keys, and any compaction or retention policy with the data-platform team.

## Create a connection

```yaml title="tapstate-work/source/lake_paimon.tapstate.yml"
version: tapstate/v1
kind: source
id: lake_paimon
connector: paimon
config:
  warehouse: s3://company-lake/paimon
  storageType: s3
  s3Endpoint: https://s3.amazonaws.com
  s3AccessKey: ${S3_ACCESS_KEY}
  s3SecretKey: ${S3_SECRET_KEY}
  s3Region: us-east-1
  database: default
mode: snapshot
tables:
  - orders
```

Use `mode: cdc` only after staging it. For target-only use, omit `mode` and `tables`.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

## Limitations

- The catalog does not define catalog/metastore fields, CDC checkpoints, schema evolution, compaction, or write conflict semantics.
- `storageType` selects conditional fields; provide only the matching S3, HDFS, or OSS inputs.
- Keep storage secrets outside committed resource files.

## Reference

| Storage | Fields |
|---|---|
| Common | `warehouse`, `storageType`, `database` (default `default`) |
| S3 | `s3Endpoint`, `s3AccessKey`, `s3SecretKey`, `s3Region`, `s3Properties` |
| HDFS | `hdfsHost`, `hdfsPort` (default `9000`), `hdfsUser` (default `hadoop`) |
| OSS | `ossEndpoint`, `ossAccessKey`, `ossSecretKey` |

`storageType` accepts `s3`, `hdfs`, `oss`, and `local`.
