## Before you begin

### Target

1. Choose the Vika space and target tables that the pipeline may write.
2. Create a credential with the minimum write scope and keep it outside source control.
3. Confirm the selected Vika plan's table, row, and API-rate limits with the account administrator.

The catalog declares `upsert` and `append` write semantics, but it does not define field mapping, conflict selection, rate-limit handling, or schema evolution behavior.

## Create a connection

```yaml title="tapstate-work/source/operations_vika.tapstate.yml"
version: tapstate/v1
kind: source
id: operations_vika
connector: vika
config:
  hostUrl: https://api.vika.cn
  spaceId: ${VIKA_SPACE_ID}
  credential: ${VIKA_CREDENTIAL}
```

Target connections have no source `mode` or `tables` section. `hostUrl` defaults to `https://api.vika.cn` in the catalog.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

Local validation checks the resource envelope and catalog-known fields. It does not authenticate to Vika, list spaces or tables, enforce a rate limit, map fields, or write a record.

## Limitations

- The catalog declares target capability only. It does not declare source reads, Snapshot, CDC, API mode, or DDL handling.
- Verify Vika API limits, table permissions, schema mapping, and conflict behavior against the deployed runtime before production use.
- Keep `credential` in an environment variable or a secret manager.
