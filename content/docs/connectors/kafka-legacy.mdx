<Aside type="caution" title="Choose the replacement for new pipelines">
  The current connection contract explicitly marks this connector as deprecated. Kafka Enhanced has a separate ID and a broader record, serializer, and Schema Registry contract. Do not change an existing resource's connector ID without staging the migration and validating downstream consumers.
</Aside>

<Aside type="note" title="Local validation boundary">
  Local validation checks the `stream` mode and recognized field shapes. It does not reach brokers, authenticate, list topics, consume records, or publish a message.
</Aside>

## Before you begin

### Source

- Allow the runtime network to reach every broker in `nameSrvAddr`.
- Grant a dedicated account read and describe access to the selected topics.
- Supply a topic name or expression in `mqTopicString`. Validate any regular expression against the real topic set before production.
- This legacy connector accepts JSON-object message strings. Decide whether malformed records should stop the read or be skipped with `kafkaIgnoreInvalidRecord`.

Kafka is a stream, not a Snapshot or CDC source. The connection does not define the initial offset policy; confirm the start position and recovery behavior with the deployed runtime.

### Target

Pre-create production topics with the required partition count, replication factor, retention policy, and write ACLs. For example:

```bash
bin/kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --replication-factor 3 \
  --partitions 3 \
  --topic orders.events
```

Grant the writer account write and describe access only to the intended topics. Choose `kafkaAcks` and `kafkaCompressionType` to match the broker policy and delivery requirements.

## Create a connection

```yaml title="tapstate-work/source/legacy_kafka.tapstate.yml"
version: tapstate/v1
kind: source
id: legacy_kafka
connector: kafka
config:
  nameSrvAddr: "broker1.internal:9092,broker2.internal:9092"
  mqTopicString: orders.events
  krb5: false
  mqUsername: ${KAFKA_USER}
  mqPassword: ${KAFKA_PASSWORD}
  kafkaSaslMechanism: PLAIN
  securityProtocol: SASL_PLAINTEXT
  kafkaIgnoreInvalidRecord: false
  kafkaAcks: "-1"
  kafkaCompressionType: lz4
  kafkaIgnorePushError: false
mode: stream
tables:
  - orders.events
```

For a target-only connection, omit `mode` and `tables`. Enable `krb5` only when the deployed runtime can provide the required keytab and Kerberos configuration fields.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

Before production, read and write one JSON object through a staging topic. Check the record with an independent consumer and verify how a malformed value, a broker restart, and a consumer restart are handled.

## Limitations

- This legacy connector supports Kafka 2.3.x and JSON-object message strings only. Use Kafka Enhanced when consumers require other native record or serialization shapes.
- Stream recovery can deliver the same record again. Use stable identifiers and idempotent downstream processing when duplicates are unacceptable.
- `kafkaIgnorePushError` can permit progress after a write error, with a risk of lost records. Do not enable it without an explicit loss-handling decision.
- Topic existence, partitions, ACLs, retention, broker security, and delivery behavior require live validation against the Kafka deployment.
