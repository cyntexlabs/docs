Use Custom Connection when the integration is defined by scripts rather than a vendor-specific connector. It is intended for controlled development and integration work where you own the script lifecycle.

## Before you begin

### Source

<SourceModeTabs>
<SourceModeTab value="snapshot">

Use **Snapshot** for a one-time script-defined read. Decide the output collection name and stable unique keys, then provide a `historyScript` that produces the expected records.

Set `syncType: initial_sync` and keep script inputs, secrets, retries, and error handling under your own change control.

</SourceModeTab>
<SourceModeTab value="snapshot-cdc">

Use **Snapshot + CDC** when you need an initial script-defined read followed by incremental work. Provide both `historyScript` and `cdcScript`, use `syncType: initial_sync+cdc`, and design an idempotent checkpoint or cursor strategy in the script contract.

The catalog cannot verify event ordering, polling cadence, cursor durability, or script output shape.

</SourceModeTab>
</SourceModeTabs>

### Target

Use a `targetScript` only after defining its input shape and failure behavior. Give the script the smallest credentials and network access it needs. The catalog declares target capability but does not describe a conflict policy, transaction boundary, or DDL behavior.

## Create a connection

```yaml title="tapstate-work/source/orders_custom.tapstate.yml"
version: tapstate/v1
kind: source
id: orders_custom
connector: custom
config:
  collectionName: orders
  uniqueKeys: id
  syncType: initial_sync+cdc
  jsEngineName: graal.js
  customBeforeOpr: false
  historyScript: ${CUSTOM_HISTORY_SCRIPT}
  cdcScript: ${CUSTOM_CDC_SCRIPT}
  customAfterOpr: false
mode: cdc
tables:
  - orders
```

For a one-time read, use `mode: snapshot`, `syncType: initial_sync`, and `historyScript`. For a target connection, omit `mode` and `tables`, then provide `targetScript` when the deployed contract requires it.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

Local validation checks the resource, mode, known field types, and enum values. It does not run the configured scripts or prove their output is compatible with a downstream pipeline.

## Limitations

- The upstream platform's script debugging, HTTP helpers, database helpers, heartbeat, and connection test are not represented as TapState capabilities here.
- The catalog does not define script sandboxing, timeout, retry, pagination, record schema, or error semantics.
- Keep scripts and secret values under review. Do not commit credentials into workspace files.

## Reference

| Field | Required | Default | Use |
|---|---|---|---|
| `collectionName` | Yes | — | Logical collection or table name. |
| `uniqueKeys` | Yes | — | Stable key field or fields used by the custom contract. |
| `syncType` | Yes | `initial_sync` | `initial_sync`, `cdc`, or `initial_sync+cdc`. |
| `jsEngineName` | Yes | `graal.js` | `graal.js` or `nashorn`. |
| `historyScript` | No | — | Script for the initial read. |
| `cdcScript` | No | — | Script for incremental work. |
| `targetScript` | No | — | Script for a target connection. |
| `customBeforeOpr`, `customBeforeScript` | `customBeforeOpr` yes | `false` | Optional pre-operation hook and script. |
| `customAfterOpr`, `customAfterScript` | `customAfterOpr` yes | `false` | Optional post-operation hook and script. |
| `batchProcess` | No | `false` | Catalog boolean for batch-oriented script handling. |
