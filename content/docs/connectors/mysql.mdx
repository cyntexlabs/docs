## Before you begin

### Source

Choose the outcome you need. Each tab is a complete preparation path; you do not need to combine steps from both tabs.

<SourceModeTabs>
<SourceModeTab value="snapshot">

Use **Snapshot** when you need one copy of the rows that exist now. TapState reads the selected tables and stops after the copy completes.

**1. Create a dedicated reader**

Restrict `%` to the TapState runtime network origin whenever possible.

```sql
CREATE USER 'tapstate_reader'@'%' IDENTIFIED BY 'your_password';
```

**2. Grant read access**

```sql
GRANT SELECT ON `your_database`.* TO 'tapstate_reader'@'%';
FLUSH PRIVILEGES;
```

Snapshot does not require MySQL Binlog or replication privileges.

</SourceModeTab>
<SourceModeTab value="snapshot-cdc">

Use **Snapshot + CDC** when the target must start with existing rows and then stay current as MySQL receives inserts, updates, deletes, and supported DDL changes.

**1. Create a dedicated CDC reader**

Restrict `%` to the TapState runtime network origin whenever possible.

```sql
CREATE USER 'tapstate_reader'@'%' IDENTIFIED BY 'your_password';
```

**2. Grant read and replication access**

```sql
GRANT SELECT ON `your_database`.* TO 'tapstate_reader'@'%';
GRANT REPLICATION SLAVE, REPLICATION CLIENT ON *.* TO 'tapstate_reader'@'%';
FLUSH PRIVILEGES;
```

**3. Enable Binlog for CDC**

CDC reads row and supported DDL events from the MySQL binary log. Configure a non-zero unique `server_id`, row-based logging, full row images, and enough retention to cover the longest planned interruption.

```ini
[mysqld]
server_id                  = 223344
log_bin                    = mysql-bin
binlog_format              = ROW
binlog_row_image           = FULL
binlog_expire_logs_seconds = 864000
```

The ten-day retention shown above is an example, not a universal requirement. If `binlog-do-db` or `binlog-ignore-db` is configured, make sure the selected database is included.

**4. Verify the active Binlog settings**

```sql
SHOW VARIABLES WHERE Variable_name IN (
  'log_bin',
  'binlog_format',
  'binlog_row_image',
  'binlog_expire_logs_seconds'
);
```

Confirm `log_bin = ON`, `binlog_format = ROW`, and `binlog_row_image = FULL`.

</SourceModeTab>
</SourceModeTabs>

### Target

Create a separate least-privilege writer. The target needs DML privileges for replicated row changes and DDL privileges only when TapState is allowed to create or evolve tables.

```sql
CREATE USER 'tapstate_writer'@'%' IDENTIFIED BY 'your_password';

-- Replicate inserts, updates, and deletes.
GRANT SELECT, INSERT, UPDATE, DELETE
  ON `analytics`.* TO 'tapstate_writer'@'%';

-- Add only when TapState may create or evolve target tables.
GRANT CREATE, ALTER, DROP
  ON `analytics`.* TO 'tapstate_writer'@'%';

FLUSH PRIVILEGES;
```

`append` and `upsert` describe how target rows are written; they are write semantics rather than connector capabilities. DDL handling is separate and covers supported create-table and column-change events. Test automatic DDL application in staging before enabling it for production pipelines.

## Create a connection

TapState stores each MySQL connection as a `.cyn.yml` file in the workspace. Create a source connection when TapState reads MySQL; create a target connection when a pipeline writes to MySQL.

### Source connection

The example below creates a Snapshot + CDC source using the current CLI contract.

```bash
cyntex new --non-interactive \
  --kind source \
  --connector mysql \
  --id orders_source \
  --mode cdc \
  --set host=db.internal \
  --set port=3306 \
  --set database=orders \
  --set username='${MYSQL_USER}' \
  --set password='${MYSQL_PASSWORD}' \
  --workdir tapstate-work
```

For a Snapshot-only source, change `--mode cdc` to `--mode snapshot`. The generated connection uses the same MySQL fields; only the read mode and CDC options differ.

Edit `tapstate-work/source/orders_source.cyn.yml` to select tables and the reading behavior:

```yaml title="tapstate-work/source/orders_source.cyn.yml"
version: cyntex/v1
kind: source
id: orders_source
connector: mysql
config:
  deploymentMode: standalone
  host: db.internal
  port: "3306"
  database: orders
  username: ${MYSQL_USER}
  password: ${MYSQL_PASSWORD}
  addtionalString: "useUnicode=yes&characterEncoding=UTF-8"
  timezone: "+00:00"
mode: cdc
tables:
  - orders
  - /customers_.*/
options:
  snapshot_mode: initial
```

| Choose | When to use it | Required MySQL setup |
|---|---|---|
| Snapshot (`mode: snapshot`) | You need one copy of the rows that exist now. | Read account with `SELECT`. |
| Snapshot + CDC (`mode: cdc`, `snapshot_mode: initial`) | You need existing rows first, followed by ongoing changes. | CDC account and binlog configuration. |
| `cdc` + `snapshot_mode: never` | You need new binlog events only. | CDC account and binlog configuration. |

Use a literal item such as `orders` for a fixed table. Use a `/regex/` item to discover matching tables; keep patterns narrow so a newly created table is not included accidentally.

#### Use a primary-replica topology

Set `deploymentMode: master-slave`, then list the primary first. Use this path instead of standalone `host` and `port`.

```yaml
config:
  deploymentMode: master-slave
  masterSlaveAddress:
    - host: mysql-primary.internal
      port: "3306"
    - host: mysql-replica-1.internal
      port: "3306"
  database: orders
  username: ${MYSQL_USER}
  password: ${MYSQL_PASSWORD}
```

For CDC from a replica, make sure it writes replicated events to its own binlog (`log_slave_updates = 1`). Check `SHOW REPLICA STATUS` (or `SHOW SLAVE STATUS` on older MySQL) and resolve replication lag before using it as a source.

### Target connection

A MySQL target is represented by the same `kind: source` connection resource, but it has no reader `mode` or `tables`. This lets a pipeline refer to it as an output connection.

```bash
cyntex new --non-interactive \
  --kind source \
  --connector mysql \
  --id analytics_mysql \
  --set host=warehouse.internal \
  --set port=3306 \
  --set database=analytics \
  --set username='${MYSQL_WRITER}' \
  --set password='${MYSQL_WRITER_PASSWORD}' \
  --workdir tapstate-work
```

The command creates `tapstate-work/source/analytics_mysql.cyn.yml`:

```yaml title="tapstate-work/source/analytics_mysql.cyn.yml"
version: cyntex/v1
kind: source
id: analytics_mysql
connector: mysql
config:
  host: warehouse.internal
  port: "3306"
  database: analytics
  username: ${MYSQL_WRITER}
  password: ${MYSQL_WRITER_PASSWORD}
```

Before using `upsert`, confirm that the target tables have the keys and column types needed to identify an existing row. If existing target tables must be preserved, verify column order and types against the source in a staging run.

Run the offline validation gate after creating or editing the connection files:

```bash
cyntex validate --workdir tapstate-work
```

## Limitations

### Source limitations

- Only tables and indexes are in scope. Temporary tables, hidden columns, triggers, views, and stored procedures are not replicated.
- CDC parses the MySQL binlog and can increase source CPU and disk I/O under a high change rate. Monitor both resources during peak workloads.
- Keep binlog files for at least seven days, and longer than the longest realistic outage or maintenance window. A CDC job cannot start or resume from a position that has expired or been deleted.
- Set `binlog_format = ROW` and `binlog_row_image = FULL`. Ensure `binlog-do-db` includes the selected database and `binlog-ignore-db` does not exclude it.
- Primary-replica addressing is supported, but TapState has not published a seamless-failover guarantee. Before a switchover, verify replication consistency, `log_slave_updates = 1` on the replica used for CDC, and acceptable replica lag.
- In MySQL 5.6 and earlier, normalize negative `TIME` values before CDC testing.
- MySQL-compatible databases can add non-native schemas or functions that the MySQL parser does not understand. Test those variants before production use.

### Target limitations

- If a target table already exists, keep its column order and data types compatible with the source or writes can fail.
- Indexes are applied during initial table creation. Later source index changes are not automatically propagated.
- Initial creation supports standard, unique, primary-key, and composite indexes. Treat full-text indexes as limited and create them explicitly when required.
- In heterogeneous pipelines, spatial, custom, `ENUM`, and `SET` values may be represented as strings. Verify the generated target schema before loading production data.
- Lower-to-higher MySQL version movement is generally safer. Higher-to-lower movement can fail on unsupported features, relaxed time formats, or invalid date values.
- DDL apply covers supported create-table and column-change events; it does not imply that later foreign-key, index, or default-value changes will be reproduced automatically.

### Consistency and conflicts

- Source and target support does not imply active-active conflict resolution. Do not update the same business keys from both sides unless the pipeline defines an external conflict policy.
- Writes made outside the replication path can cause source-target differences.
- Avoid DDL on participating source tables during a snapshot. Schema changes can invalidate the snapshot or cause it to fail.
- Validate row counts and keyed samples after a snapshot, then test `INSERT`, `UPDATE`, and `DELETE` events before opening the target to consumers.

## Reference

### Connection settings

| Field | Required | Default | Use |
|---|---|---|---|
| `deploymentMode` | No | `standalone` | Select `standalone` for one server or `master-slave` for a primary-replica list. |
| `host` | With `standalone` | — | MySQL hostname or IP address. |
| `port` | With `standalone` | `3306` | MySQL service port. It is a string in the catalog. |
| `masterSlaveAddress` | With `master-slave` | — | Ordered array of `{ host, port }`; put the primary first. |
| `database` | Yes | — | Database name. Create separate connections for separate databases. |
| `username` | Yes | — | Database account name. |
| `password` | No | — | Database password. Supply this from secret-aware environment handling. |
| `addtionalString` | No | `useUnicode=yes&characterEncoding=UTF-8` | Additional connection parameters. The inherited spelling is exact and must not be corrected to `additionalString`. |
| `timezone` | No | empty | UTC offset for timezone-naive values, for example `+00:00` or `+08:00`. |

### Data compatibility

The reused connector baseline covers strings, integer and decimal types, date/time values, binary objects, spatial values, `BIT`, `ENUM`, `SET`, and `JSON`.

- DML replication covers `INSERT`, `UPDATE`, and `DELETE`.
- DDL capture and apply cover adding, changing, renaming, and dropping columns.
- Test heterogeneous pipelines with spatial, custom, `ENUM`, and `SET` values because downstream systems commonly represent them as strings.
