Use Lark Task to create Lark or Feishu tasks from pipeline records.

<Aside type="note" title="Local validation boundary">
  Local validation checks the resource and application credential field shapes. It does not authenticate to Lark, resolve assignees, create tasks, or verify application permissions.
</Aside>

## Before you begin

### Target

1. Create or select a self-built Lark/Feishu application.
2. Copy the App ID and App Secret into a secret manager.
3. Set the app's availability scope to include the task owners and followers that the pipeline will reference.
4. Publish the app and grant the task permissions required by the tenant administrator.

Pipeline records should supply the business fields needed to create a task, such as title, description, assignee, followers, due time, and an optional link.

## Create a connection

```yaml title="tapstate-work/target/tasks_lark.tapstate.yml"
version: tapstate/v1
kind: target
id: tasks_lark
connector: lark-task
config:
  app_id: ${LARK_APP_ID}
  app_secret: ${LARK_APP_SECRET}
```

Target connections do not have a source `mode` or `tables` section. Keep the App Secret in an environment variable or secret manager.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

Create one task in a test tenant before enabling production records. Verify the app can resolve the selected owner and that due-time values use the expected tenant timezone.

## Limitations

- Assignee visibility, application publication, task permission, and API rate limits are enforced by Lark/Feishu at runtime.
- The connector is for task creation; it does not provide a documented source read mode.
- Field mapping from pipeline records to task fields must be tested with representative data.

## Reference

| Field | Required | Use |
|---|---|---|
| `app_id` | Yes | Self-built application App ID. |
| `app_secret` | Yes | Application secret; inject it through an environment variable. |
