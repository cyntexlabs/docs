## Before you begin

### Target

Create or select destination indices and an account with only the index privileges required by the pipeline. Decide mapping, analyzers, and lifecycle policy before the first production write; those settings are enforced by Elasticsearch rather than the local resource validator.

## Create a connection

```yaml title="tapstate-work/source/search_elasticsearch.tapstate.yml"
version: tapstate/v1
kind: source
id: search_elasticsearch
connector: elasticsearch
config:
  host: elasticsearch.internal
  port: "9200"
  user: ${ELASTICSEARCH_USER}
  password: ${ELASTICSEARCH_PASSWORD}
  sslValidate: true
  validateCA: true
  sslCA: ${ELASTICSEARCH_CA_PEM}
```

The connection requires `sslCA`; supply the CA material through a secret-aware variable and confirm the runtime's expected PEM/path representation before deployment.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

This checks local field shape only. It does not test DNS, TLS negotiation, certificate trust, authentication, index permissions, mappings, or writes.

## Limitations

- TapState supports a target only; it does not declare source reads, Snapshot, CDC, or DDL behavior.
- The connector covers Elasticsearch 7.6. Version compatibility and mappings need staging verification.
- The connection exposes certificate-related fields but not a complete security model. Do not assume defaults for HTTPS, certificate paths, or hostname verification.

## Reference

| Field | Required | Use |
|---|---|---|
| `host` | Yes | Elasticsearch host. |
| `port` | Yes | Elasticsearch service port; field type is string. |
| `user` | No | Account name. |
| `password` | No | Account password; use an environment variable. |
| `sslValidate` | No | Boolean for SSL validation. |
| `validateCA` | No | Boolean for CA validation. |
| `sslCA` | Yes | CA material required by the current connection contract. |
