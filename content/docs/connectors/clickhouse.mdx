## Before you begin

### Source

Create a least-privilege reader for the selected database:

```sql
CREATE USER tapstate_reader IDENTIFIED WITH sha256_password BY 'replace_with_a_secret';
GRANT SELECT ON analytics.* TO tapstate_reader;
```

### Target

Create a writer. Add table-creation and alteration permissions only when pipelines are allowed to manage destination tables.

```sql
CREATE USER tapstate_writer IDENTIFIED WITH sha256_password BY 'replace_with_a_secret';
GRANT INSERT ON analytics.* TO tapstate_writer;
```

## Create a connection

```yaml title="tapstate-work/source/analytics_clickhouse.tapstate.yml"
version: tapstate/v1
kind: source
id: analytics_clickhouse
connector: clickhouse
config:
  host: clickhouse.internal
  port: "8123"
  database: analytics
  user: ${CLICKHOUSE_USER}
  password: ${CLICKHOUSE_PASSWORD}
mode: snapshot
tables:
  - events
```

For a target-only connection, omit `mode` and `tables`.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

This checks the local resource, Snapshot mode, and catalog-known field types—not network access, ClickHouse privileges, or query results.

## Limitations

- The catalog declares Snapshot only; it does not declare a CDC or DDL contract.
- ClickHouse HTTP is commonly served on port `8123`; confirm the endpoint and any encryption settings with your deployment rather than assuming a TLS field.
- Test target table engines, sort keys, and data-type mappings with representative data before production writes.
