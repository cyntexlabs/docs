## Before you begin

### Source

<SourceModeTabs>
<SourceModeTab value="snapshot">

Use **Snapshot** when you need the rows that exist now and do not need later changes.

```sql
CREATE USER 'tapstate_reader'@'%' IDENTIFIED BY 'replace_with_a_secret';
GRANT SELECT ON `orders`.* TO 'tapstate_reader'@'%';
FLUSH PRIVILEGES;
```

Snapshot does not require replication privileges or binary logging.

</SourceModeTab>
<SourceModeTab value="snapshot-cdc">

Use **Snapshot + CDC** when the target must begin with existing rows and then receive inserts, updates, and deletes.

```sql
CREATE USER 'tapstate_reader'@'%' IDENTIFIED BY 'replace_with_a_secret';
GRANT SELECT ON `orders`.* TO 'tapstate_reader'@'%';
GRANT REPLICATION SLAVE, REPLICATION CLIENT ON *.* TO 'tapstate_reader'@'%';
FLUSH PRIVILEGES;
```

Enable row-based binary logging and retain logs longer than the longest credible pipeline interruption:

```ini
[mysqld]
server_id       = 223344
log_bin         = mariadb-bin
expire_logs_days = 10
max_binlog_size = 100M
binlog_format   = ROW
binlog_row_image = FULL
```

`server_id` must be unique among servers and replication clients. Set `expire_logs_days` to a recovery window longer than the longest expected interruption, and size `max_binlog_size` for your write volume.

Restart MariaDB during an approved maintenance window, then verify the effective settings:

```sql
SHOW VARIABLES LIKE 'log_bin';
SHOW VARIABLES LIKE 'binlog_format';
SHOW VARIABLES LIKE 'binlog_row_image';
SHOW BINARY LOGS;
```

</SourceModeTab>
</SourceModeTabs>

### Target

Create a dedicated writer. Grant table-creation or alteration privileges only when the destination schema is allowed to be managed by a pipeline.

```sql
CREATE USER 'tapstate_writer'@'%' IDENTIFIED BY 'replace_with_a_secret';
GRANT SELECT, INSERT, UPDATE, DELETE,
      ALTER, CREATE, CREATE ROUTINE, CREATE TEMPORARY TABLES, DROP
ON `analytics`.* TO 'tapstate_writer'@'%';
FLUSH PRIVILEGES;
```

Remove `ALTER`, `CREATE`, routine, temporary-table, and `DROP` privileges when the pipeline must write only to existing tables.

## Create a connection

```yaml title="tapstate-work/source/orders_mariadb.tapstate.yml"
version: tapstate/v1
kind: source
id: orders_mariadb
connector: mariadb
config:
  host: mariadb.internal
  port: "3306"
  database: orders
  username: ${MARIADB_USER}
  password: ${MARIADB_PASSWORD}
  addtionalString: "useUnicode=yes&characterEncoding=UTF-8"
  timezone: "+00:00"
mode: cdc
tables:
  - orders
```

For a one-time copy, use `mode: snapshot`. For a target connection, omit `mode` and `tables`, and use the writer credentials.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

Local validation checks the resource, mode, and catalog-known field types. It does not test network access, credentials, MariaDB privileges, binlog retention, or data movement.

## Limitations

- The catalog declares `snapshot` and `cdc`, but does not declare DDL handling; this page does not claim DDL capture or application.
- The upstream preparation baseline is MariaDB 10.x. Test versions, replication topology, and data types with representative data before production use.
- CDC requires `ROW` binary logging with a full row image. Statement- or mixed-format logs do not provide the documented change record.
- Expired binary logs break resume from the saved position. Monitor log retention and free disk space instead of relying only on a fixed number of days.
- `server_id` collisions can disrupt replication clients. Assign a stable, unique value in every environment.
- The catalog does not expose transport-encryption fields. Do not infer a TapState TLS configuration from server-side MariaDB settings.
