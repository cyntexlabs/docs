## Before you begin

Have a Salesforce administrator approve the connected account and least-privilege object access. Ensure the authorization value is stored in a secret manager or environment variable.

## Create a connection

```yaml title="tapstate-work/source/revenue_salesforce.tapstate.yml"
version: tapstate/v1
kind: source
id: revenue_salesforce
connector: salesforce
config:
  code: ${SALESFORCE_AUTHORIZATION_CODE}
mode: api
```

Confirm authentication expiry, refresh, object selection, and incremental behavior with the deployed runtime before using this connection for production data.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

Local validation checks the API mode and `code` field shape only. It does not authenticate, test scopes, enumerate objects, or read CRM records.

## Limitations

- The catalog declares API source only; it does not declare target writes, Snapshot, CDC, DDL, or rate-limit behavior.
- The old UI's authorization workflow is upstream baseline evidence, not a TapState product claim.
