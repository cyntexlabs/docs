<Aside type="note" title="Authorization">
  The connection requires `code`. Obtain it through the authorization flow supplied by your TapState deployment; local validation does not exchange or refresh it.
</Aside>

## Before you begin

### Source

Choose the HubSpot account and objects to expose, then arrange administrator-approved authorization with the smallest scopes needed. Store the resulting credential outside source control.

## Create a connection

```yaml title="tapstate-work/source/marketing_hubspot.tapstate.yml"
version: tapstate/v1
kind: source
id: marketing_hubspot
connector: hubspot
config:
  code: ${HUBSPOT_AUTHORIZATION_CODE}
mode: api
```

TapState does not expose a token-refresh field, selected-object field, or account selector. Confirm those runtime behaviors before production use.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

<ValidationStatusGuide />

Local validation does not authorize HubSpot, verify scopes, enumerate CRM objects, or read records.

## Limitations

- API source only; no target, Snapshot, CDC, DDL, or API-rate contract is declared.
- Use only the documented resource fields. Do not add OAuth or account settings that are not accepted by the connection schema.

## Reference

| Field | Required | Use |
|---|---|---|
