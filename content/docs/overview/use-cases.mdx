TapState is designed for operational data that loses value when it becomes stale. Each pattern below describes the customer outcome and what the current release covers.

<Aside type="note" title="Release status">
  The current release ships the authoring layer for these patterns; live execution arrives with the runtime. See [Release Status](/docs/overview/release-status).
</Aside>

## Core-system offloading

**Problem:** Applications, dashboards, and AI systems query a production database directly, increasing coupling and operational risk.

**With TapState:** Capture source changes and maintain a read-oriented copy or materialized view for downstream consumers — the source database is never queried during steady-state processing.

**Available today:** Define source, pipeline, transform, and target intent as validated `.tapstate.yml` resources.

## Database migration and coexistence

**Problem:** A new system must be prepared while the old system remains active.

**With TapState:** Start with a snapshot, continue with CDC, validate consistency, and cut over once the target is current.

**Available today:** Connector preparation guides, snapshot/CDC modes, target connection resources, reference closure, and offline config validation.

## Current customer or account views

**Problem:** Account, order, inventory, and entitlement data is fragmented across systems, while applications need one current view.

**With TapState:** Capture changes, join or reshape records in flight, and materialize current operational state as always-fresh views.

**Available today:** Declarative transform and view resource models plus schema validation.

## Operational context for AI

**Problem:** Retrieval from documents or a warehouse can omit the latest system-of-record state — and agents that act on stale data act wrongly.

**With TapState:** Provide governed, queryable operational state to agents and applications, while keeping model choice separate from the data platform.

**Available today:** AI-readable docs, JSON Schema, non-interactive scaffolding, coded diagnostics, and structured CLI output.

## Event and analytics feeds

**Problem:** Downstream teams need a clean stream without independently reading the same production database.

**With TapState:** Capture once, apply common transformations, and route approved records to Kafka or other targets.

**Available today:** Catalog-backed Kafka authoring and pipeline intent.

## Evaluating a use case for production

Before treating a pattern as production-ready, verify four layers separately:

1. **Connector contract:** ID, mode, fields, permissions, and source-system preparation.
2. **Authoring contract:** valid resources and closed references.
3. **Runtime behavior:** connectivity, snapshots, CDC, transformations, checkpoints, and recovery.
4. **Operational evidence:** data correctness, latency, source impact, security, and failure handling in the intended environment.

The current release directly covers the first two layers; the [Roadmap](/docs/overview/roadmap) tracks the rest.
