> **Kafka moves events. Warehouses analyze history. TapState turns database truth into live operational state.**

TapState is an open-source data platform built around one idea: **Capture. Transform. Serve. One deployable.** It taps your production databases with log-based change data capture, reshapes change streams in flight, and serves continuously fresh, queryable state to applications, APIs, and AI agents — one product boundary instead of a four-system assembly project.

## The problem: the assembly project

Keeping operational data fresh today usually means assembling and operating separate capture, broker, processing, and serving systems:

<DataPathComparison />

Each layer in the assembled stack is good software. The cost is the glue: offsets, retries, schema drift, serialization, upgrades, and dashboards duplicated across every hop — an integration project that exists only to move your own data.

Composable stacks remain the right choice for some workloads; [TapState and the streaming stack](/docs/overview/vs-streaming-stack) gives an honest decision guide. TapState targets the common case: teams whose dominant problem is keeping operational state current, not building bespoke stream topologies.

## One data path, three depths

TapState is not three features — it is one road with three depths of adoption. Most teams enter at Capture; each stage makes the next one a configuration change instead of a new system.

### Capture — CDC you can bet the database on

Log-based change capture from transaction logs (MySQL binlog, PostgreSQL WAL, MongoDB oplog, and more), plus snapshot, stream, API, and file modes — all behind connector-defined contracts. Initial load and continuous sync are the same pipeline, not two tools.

### Transform — reshape data in flight, not in batch

Filter, rename, enrich, join, and route change streams inside the same engine, so downstream consumers receive answer-shaped data instead of raw tables plus a to-do list. No separate processing cluster to size, deploy, and babysit.

### Serve — fresh state, queryable now

Pipelines terminate in an endpoint, not another queue: materialized, always-current views of your business — accounts, orders, inventory, entitlements — served to applications and agents, and pushed onward to Kafka, lakes, or webhooks when other teams need the stream.

## Who it's for

TapState is built for operational data that loses value when it goes stale: fintech operations and fraud/risk, e-commerce inventory, logistics tracking, operational-cache replacement, cross-system customer/account views, and internal copilots or AI agents that need the latest system-of-record state. [Use Cases](/docs/overview/use-cases) walks through each pattern.

## Why fresh state matters more now

For thirty years the consumer of enterprise data was a human reading a dashboard, and humans tolerate staleness. AI moves consumption to software that *acts* — agents approving refunds, adjusting inventory, answering customers. When the consumer acts instead of reads, stale data doesn't mean an outdated chart; it means wrong actions. TapState is not "AI-native" — it is the infrastructure that AI adoption stresses. Vector search gives agents semantic memory; TapState's job is committed business truth.

## Provenance

TapState is built for teams that need a reviewable, declarative path from operational changes to continuously fresh data. It combines an offline authoring workflow with connector contracts and runtime-oriented resource definitions, without requiring a separate stack of bespoke integration assets.

## The authoring contract in these docs

The documentation describes a reviewable authoring layer built from `.tapstate.yml` resources, connector contracts, and deterministic validation. Its current executable availability is tracked separately in [Release status](/docs/overview/release-status).

- **A reviewable data contract.** Sources and pipelines are ordinary `.tapstate.yml` files that work with Git review, environment-variable substitution, and automated generation.
- **Deterministic guardrails.** The documented `new → edit → ls / desc / explain → validate` loop checks resource structure, reference closure, connector configuration rules, and mode combinations.
- **Connector-aware authoring.** The connector catalog supplies the exact field and mode contract for each guide.
- **Built for people and agents.** JSON Schema, non-interactive scaffolding, structured output, and [`llms.txt`](/llms.txt) keep the same contract readable to humans and AI assistants.
