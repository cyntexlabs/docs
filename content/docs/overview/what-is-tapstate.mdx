TapState is designed to keep operational data useful after it leaves the system of record. It brings capture, in-flight transformation, and delivery into one governed data path so teams can define the outcome without assembling a different control model for every hop.

<Aside type="note" title="Product direction and release evidence are separate">
  This page explains the product model. It does not establish that every runtime layer is available in a published TapState release. Use [Release status](/docs/overview/release-status) to understand the current evidence boundary.
</Aside>

## The problem TapState addresses

Keeping operational data current often requires separate capture, broker, processing, and serving systems:

```text
Source systems → Capture → Broker → Processing → Serving store → Apps and agents
```

Each component can be a sound choice. The operational cost comes from the seams: duplicated configuration, offset ownership, schema evolution, serialization, retries, access control, and observability across multiple products.

TapState focuses on a narrower customer outcome: keep selected operational state current and usable without turning every data path into a custom streaming platform.

## One path, three responsibilities

### Capture

Read an initial state and, when the connector contract allows it, continue from transaction logs, WAL, Oplog, changelogs, message streams, files, or APIs. The connector guide defines the supported role, mode, preparation, and limitations.

### Transform

Filter, reshape, enrich, and route records as they move through the data path. A declarative resource keeps those decisions reviewable alongside the source and destination intent.

### Serve

Deliver useful state to a destination, event consumer, application, or agent-facing interface. The serving shape should follow the consumer's access pattern rather than expose raw source tables by default.

## Where TapState fits

TapState is a candidate when:

- applications need a current read model without querying the production database;
- a migration requires an initial copy followed by continuous changes;
- multiple consumers need the same governed operational data;
- teams want connector, transformation, and destination intent in reviewable resources;
- agents need recent system-of-record state with deterministic product controls.

A composable Kafka and Flink stack remains appropriate for complex event-time processing, established organization-wide streaming platforms, or independent scaling of every layer. See [TapState and the streaming stack](/docs/overview/vs-streaming-stack) for the tradeoffs.
