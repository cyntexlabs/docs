This page defines how to interpret product claims while the documentation and implementation are converging on a TapState-branded release baseline.

<Aside type="caution" title="No release baseline is asserted here">
  The latest implementation and product rebrand have not yet been established as the evidence baseline for this site. Do not interpret a command, roadmap item, connector page, or target-architecture description as proof that a public TapState artifact ships that behavior.
</Aside>

## Evidence levels

| Level | What it means | What it does not prove |
|---|---|---|
| Documented intent | The product model or desired workflow is explained | Implementation or availability |
| Code-backed contract | A current schema, Catalog entry, or implementation path declares the behavior | That the artifact you have includes it or that it works end to end |
| Artifact-verified | The exact command or resource was exercised against an identified TapState artifact | External connectivity or production behavior unless the test included it |
| Runtime-verified | The identified deployment completed the operation with representative data | Universal performance or compatibility across environments |
| Published release contract | Versioned release notes, artifacts, and installation instructions make the behavior public | Suitability for a workload without environment-specific validation |

## Current documentation boundary

The documentation can currently establish:

- the intended Capture–Transform–Serve product model;
- a canonical reader and AI documentation structure;
- connector preparation inherited from reviewed upstream sources;
- connector IDs, fields, modes, and target capability from the selected Catalog snapshot;
- the documented TapState resource naming contract.

The following still require alignment with the next implementation baseline:

- executable name, version output, and packaged command set;
- resource filenames, API version, defaults, and validation behavior;
- installation methods and public artifacts;
- live connection, discovery, pipeline execution, recovery, and serving behavior;
- Web, API, and MCP surfaces;
- release maturity, roadmap phase, and performance claims;
- website links and download paths.

## How pages should express status

- Product and concept pages describe the durable problem, model, and decision boundaries.
- Connector pages distinguish upstream preparation, Catalog-backed fields, and deployment verification.
- Quickstarts label commands as a documentation contract until an identified artifact is exercised.
- Reference pages must not present preview syntax as a shipped resource contract.
- Performance and reliability numbers require a reproducible runtime measurement.

## Before a public release update

The documentation team should verify one identified implementation commit and artifact, rerun every executable example, refresh the connector Catalog, check public installers and packages, and reconcile the website with the docs. Until that work is complete, use conservative language and ask the deployment owner for its actual contract.
