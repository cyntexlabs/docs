TapState v1 is delivered in four phases. Each phase is independently demo-able, runnable against real data, and covered by end-to-end tests. [Release Status](/docs/overview/release-status) tracks exactly what is shipped at any moment.

## Release Phases

### POC <Badge text="Current" variant="tip" />

**Goal:** a runnable skeleton — a single-node CDC pipeline works end-to-end.

| Deliverable | Status |
|---|---|
| Maven mono-repo skeleton with enforced architecture boundaries | ✅ Done |
| Native CLI tooling (GraalVM 21, ~10ms startup) | ✅ Done |
| Full DSL grammar (model / parser / schema) | 🔄 In progress |
| Connector catalog build pipeline (60+ connectors) | 🔄 In progress |
| Offline CLI (`new` / `validate` / `explain`, native binary) | 🔄 In progress |
| Connector loading + single-node basic CDC (MySQL → MongoDB) | ⏳ Not started |
| Control core (CRUD + lifecycle) + basic auth | ⏳ Not started |

**Exit criteria:** register a connector, run a YAML single-table task (full snapshot / basic CDC) end-to-end, and query its status.

---

### Alpha (~4 weeks, soft release)

**Goal:** a real sync pipeline with basic processing, demoable end-to-end; minimal AI integration.

| Deliverable |
|---|
| CDC connectors (directly included) |
| Basic processing nodes (rename / JavaScript / typeFilter / unwind / date / row-level filter) |
| **Minimal MCP** (read-only + scaffold/explain; bring-your-own agent) |
| End-to-end test framework (declarative YAML specs) |
| OpenTelemetry metrics and logging |

**Exit criteria:** a sync task with basic processing operators runs end-to-end.

---

### Beta (~8 weeks, OSS core complete)

**Goal:** distributed CDC, joins and merges, orchestration, and monitoring — CLI + basic UI dual frontend.

| Deliverable |
|---|
| Distributed cluster (quorum + heartbeat) |
| Distributed CDC with shared log mining, offset persistence, and resume |
| Cache management (two-level in-memory cache) |
| **DuckDB join node** |
| **Master-detail merge operator (MERGE family)** |
| DDL / schema-drift handling |
| Multi-task dependency orchestration (task-of-tasks) |
| **Complete MCP toolset** (mutable CRUD + full lifecycle operations) |
| **Basic UI (open source)** |
| OpenLineage field-level lineage |

---

### GA (~8 weeks, enterprise-ready)

**Goal:** external API, external data lake, automatic HA, and enterprise plugin integration.

| Deliverable |
|---|
| **API server with API management** — open source |
| Custom processing nodes |
| Debezium integration |
| **Paimon incremental data lake** (external storage) |
| Automatic task failover (HA) |
| Containerization and port consolidation |
| Enterprise UI |
| **Enterprise plugins (closed-source)**: RBAC / LDAP / lineage & tracing / enterprise connectors (DB2, Sybase, SQL Server, PostgreSQL, Oracle agents) |

## Key Design Decisions

Locked invariants, tracked as architecture decision records in the source repository:

- **YAML-only authoring**: SQL is not an authoring format; it appears only as an embedded value inside DuckDB nodes
- **Bring your own agent, no bundled LLM**: the MCP server contains no model
- **No auto-fix**: AI monitoring/alerting is read-only; mutable operations require a human in the loop
- **Open core**: no license gating; closed-source functionality ships as SPI plugins
- **Single binary + `--role`**: merged deployment for small scale; role-separated for large scale
