## Hot Data vs Cold Data

TapState is purpose-built for **hot data** — operational, real-time data that powers customer-facing applications and AI agents. This is distinct from the cold data stored in warehouses and lakes:

| | TapState | Warehouse / Lake |
|---|---|---|
| **Data temperature** | Hot (fresh to the latest commit) | Cold (minutes to hours old) |
| **Primary use** | Apps, APIs, AI agents, fraud detection | Reports, ML training, historical analysis |
| **Write pattern** | Continuous CDC stream | Batch loads |
| **Query pattern** | Point queries, live views | Aggregation, scans, joins |

TapState does not replace your data warehouse — it complements it. The warehouse stores history for analysis; TapState maintains the current operational state for action.

## The Streaming-First Principle

Unlike batch ETL tools that process data in windows (hourly, nightly), TapState processes each change event the moment it is captured — not micro-batching, but true event-by-event streaming. A committed change flows from the source's transaction log, through enrichment, into the Materialized View Store, and out through the Query Context Server as one continuous motion.

Batch-centric thinking asks: "What changed since last night?" TapState asks: "What is changing right now?"

## Three-Tier Storage Architecture

### L1 — Distributed In-Memory (Hazelcast)

The hot tier. Hosts the **Change Log Ringbuffer** — the central in-memory buffer through which all captured change events flow — plus task runtime state and active CDC streams. It fans out to Transform and Materialize consumers without re-reading the source.

### L2 — Persistent Source of Truth (MongoDB)

The operational persistence tier. Everything that must survive a process restart lives here: connection definitions and pipeline configurations, CDC offsets and checkpoints (for resume and replay), task HA leases, and connector distribution. MongoDB is the **source of truth**; the in-memory tier is derived from it on startup.

### L3 — Incremental Data Lake (Paimon, GA phase)

An external add-on for long-term, queryable audit history — bridging TapState's hot operational data and a finance-grade archival layer.

Implementation details of the in-memory tier and the HA design live in [Engineering Internals](/docs/reference/internals).

## The Lookup Cache: Streaming Joins Without Querying the Source

One of the most critical patterns in real-time processing is **stream enrichment** — joining a live change event with reference data from another table. Example: as a `Transaction` event flows through, attach the matching `Customer.loyaltyTier` so the downstream consumer has full context.

TapState solves this with the **Lookup Cache** (L1, in-memory), kept in sync with the **Source Replica Store**:

```text
Change Log Buffer ──→ Transform (join lookup via Lookup Cache)
        │
        └──→ Source Replica Store (keeps Lookup Cache fresh)
```

The source database is **never queried** during steady-state processing. The Lookup Cache provides reference data entirely in memory, enabling low-latency joins at scale.

## DSL Artifact Lifecycle

The server architecture keeps an authoring copy and a canonical stored copy:

```text
Local .tapstate.yml file     TapState resource store
(authoring draft)    ──→     (official store, single source of truth)
                    apply
```

The server workflow validates and canonicalizes a local resource before storing it by ID; export and diff operations round out the lifecycle. The local file is an authoring draft; the engine reads the canonical stored resource. These server-backed operations arrive with the runtime.
