TapState's storage model is organized by responsibility rather than a promise about one database technology. A data path needs fast working state, durable recovery state, and consumer-facing materialized state.

<Aside type="note" title="Target storage model">
  Concrete storage engines, retention defaults, replication, and high-availability behavior require an identified implementation and release contract.
</Aside>

## Operational state and analytical history

Operational consumers usually need the latest valid state of an account, order, inventory item, or entitlement. Analytical systems usually retain a longer history optimized for scans, aggregation, and model training.

| Concern | Operational state | Analytical history |
|---|---|---|
| Primary question | What is true now? | What happened over time? |
| Common access | Key lookup, current view, event reaction | Scan, aggregate, compare periods |
| Update pattern | Continuous changes and deletes | Batch or append-oriented ingestion |
| Failure sensitivity | Stale state can affect an action | Delayed data usually affects analysis timing |

The two are complementary. A production design can maintain current state for applications while retaining approved history in a warehouse or lake.

## Working state

The data plane needs short-lived state for active processing, buffering, routing, and lookups. Working state improves throughput, but it must not be the only copy of information required after restart.

## Recovery state

Durable recovery state includes:

- connector positions, offsets, or checkpoints;
- pipeline identity and canonical configuration;
- retry and failure context;
- schema and mapping state;
- ownership information needed to prevent conflicting execution.

Recovery semantics should answer whether a restart resumes, replays, skips, or redelivers records. Connector retention must be long enough for the saved position to remain usable.

## Materialized state

Materialized state is the current consumer-facing result. It may live in a target database, search system, warehouse, lakehouse, or future serving layer.

Its correctness depends on:

- stable keys for updates and deletes;
- defined append or update behavior;
- schema-change policy;
- ordering and duplicate handling;
- reconciliation with the system of record;
- retention and deletion requirements.

## Audit history

An audit or replay store answers different questions from a current view. Before retaining change history, define the legal basis, sensitive-field handling, retention period, access policy, and deletion process.

Do not infer long-term history, replay, exactly-once behavior, or a specific storage engine from this concept page. Verify those behaviors against the deployed runtime and its published architecture.
