MCP is a possible client interface for a future TapState control plane. It would allow an AI assistant to call published product operations while TapState remains responsible for validation, authorization, runtime state, and audit behavior.

<Aside type="caution" title="No endpoint or toolset is published by this page">
  Server URL, transport, authentication, tool names, scopes, and lifecycle behavior require a TapState release contract. Do not use examples from an earlier product name or an internal design as deployment instructions.
</Aside>

## Intended boundary

```text
AI client
   │  MCP request with explicit scope
   ▼
TapState control interface
   │  shared validation, authorization, and audit
   ▼
Resource and runtime operations
```

The MCP server would contain no bundled language model. Users choose the client and model; TapState exposes deterministic operations and their results.

## Candidate operation groups

| Group | Examples of intent | Required safeguards |
|---|---|---|
| Documentation | Explain a field or connector | Versioned canonical content |
| Discovery | List resources or inspect a source schema | Read scope and sensitive-field filtering |
| Authoring | Scaffold or validate a resource draft | Deterministic schema and no implicit apply |
| Observation | Read status, lag, or diagnostics | Deployment identity and read authorization |
| Mutation | Apply, start, stop, or delete | Explicit target, write scope, audit, and confirmation |

These are design categories, not a promised list of MCP tools.

## Client safety requirements

- Display the TapState deployment and resource before a mutation.
- Keep credentials out of prompts and generated resource files.
- Require explicit authorization for destructive or lifecycle operations.
- Return structured errors that separate validation, authorization, and runtime failures.
- Do not let an assistant treat documentation as live deployment state.
- Record enough audit context to identify the client, operation, target, and result.
