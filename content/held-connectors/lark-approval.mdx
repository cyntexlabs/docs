## Before you begin

### Source

1. Create or select a Lark/Feishu self-built application.
2. Copy the App ID and App Secret from the application's credentials page.
3. Have the tenant administrator grant the application the approval-data permissions required by the intended workflow.
4. If your deployment requires them, obtain the approval `code` and `definitionCode` from the administrator or platform API tooling.

![Locate application credentials in Lark or Feishu Open Platform](/images/connectors/lark-app-credentials.png)

Keep the secret outside source control. This connector uses application credentials supplied in the connection resource and does not start an interactive authorization flow.

## Create a connection

```yaml title="tapstate-work/source/purchase_approvals.tapstate.yml"
version: tapstate/v1
kind: source
id: purchase_approvals
connector: lark-approval
config:
  app_id: ${LARK_APP_ID}
  app_secret: ${LARK_APP_SECRET}
  code: ${LARK_APPROVAL_CODE}
  definitionCode: ${LARK_APPROVAL_DEFINITION_CODE}
mode: api
```

Use `code` or `definitionCode` only when the Lark/Feishu administrator provides them for the approval scope you need; otherwise omit them.

## Validate the configuration

```bash
tapstate validate --workdir tapstate-work
```

## Limitations

- This connector reads approval data only. It does not write records or apply schema changes.
- Lark/Feishu enforces application visibility, tenant policy, and API rate limits at runtime.
- Do not commit App Secret values in a `.tapstate.yml` file.

## Reference

| Field | Required | Use |
|---|---|---|
| `app_id` | Yes | Self-built application App ID. |
| `app_secret` | Yes | Application secret; inject through an environment variable. |
| `code` | No | Approval code supplied for the selected approval scope. |
| `definitionCode` | No | Approval definition code supplied for the selected workflow. |
